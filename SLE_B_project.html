문서 처리 파이프라인 관리자가 문서를 업로드할 때 실행
1-a 파일 업로드 & 파싱
▼

PDF, DOCX, 이미지 파일을 받아 텍스트를 추출. 카테고리 분류 용도로 첫 페이지만 먼저 추출하고, 전체 내용은 이후 단계에서 처리.

PDF → pdfplumber, DOCX → python-docx, 이미지 → Gemini Vision
HWP 미지원 — PDF로 변환 후 업로드 안내
첫 페이지만 추출 → 카테고리 분류 비용 절감
1-b 카테고리 초안 생성
Gemini API
▼

첫 페이지 텍스트를 Gemini에게 넘겨 대카테고리 + 소카테고리 구조 초안을 JSON으로 생성. 문서 종류에 따라 동적으로 결정되므로 사내 규정 외 다른 도메인에도 활용 가능.

대카테고리 자동 제안 (예: 복리후생, 인사노무, 보안정책)
각 대카테고리의 소카테고리까지 함께 생성
JSON 스키마 강제로 파싱 안정성 확보
1-c 관리자 카테고리 확정
▼

LLM이 제안한 카테고리 초안을 관리자에게 보여주고 수정/확정. LLM 오분류를 사람이 잡아주는 단계. 확정 후 폴더 구조 자동 생성.

UI에서 카테고리명 수정, 소카테고리 추가/삭제 가능
확정 시 storage/ 하위 폴더 구조 자동 생성
categories.json에 구조 저장
1-d 문서 분류 및 저장
Gemini API
▼

전체 문서를 읽고 각 소카테고리에 해당하는 내용을 추출해서 저장. 조항 번호·항목 번호 등 원본 구조를 그대로 보존. 한 문서가 여러 카테고리에 걸치면 내용을 분리해서 각각에 저장.

소카테고리별로 관련 내용 추출 → {카테고리}/{소카테고리}.txt에 저장
제N조, ①② 등 원본 조항 구조 보존 (출처 표시용)
문서 업데이트 시 해당 파일 덮어쓰기 (버전 관리 없음, 프로토타입)
SSE로 진행 상황 프론트엔드에 실시간 전달
질문 처리 파이프라인 사용자가 질문할 때마다 실행
2-a 대카테고리 우선순위 분류
Gemini API
▼

사용자 질문과 대화 기록을 바탕으로 각 대카테고리에 우선순위 점수를 매김. 점수 3(관련 없음)은 탐색에서 완전히 제외해 불필요한 LLM 호출 차단.

0: 가능성 높음 → 1: 가능성 있음 → 2: 낮음 → 3: 제외
score 0, 1만 탐색 대상, 오름차순으로 순회
2-b 소카테고리 우선순위 분류
Gemini API
▼

선택된 대카테고리 안에서 소카테고리 탐색 순서를 결정. 관련 없는 소카테고리는 건너뜀.

질문과 관련 높은 소카테고리부터 탐색
관련 없는 소카테고리는 리스트에서 제외
2-c 파일 읽기 & Found 판단
Gemini API
▼

소카테고리 txt 파일을 읽어 Gemini에게 넘기고, 질문에 답할 수 있는 내용이 있는지 판단. 있으면 관련 조항 원문과 함께 found=true 반환.

found=true → 즉시 탐색 종료, 답변 생성으로 이동
found=false → 다음 소카테고리로 이동
탐색 경로(path_taken) 매 단계마다 기록
✓ FOUND
관련 조항 추출 → 답변 생성 단계로
✗ NOT FOUND
다음 소카테고리 → 소진 시 다음 대카테고리 → 전체 소진 시 Fallback
2-d 답변 생성
Gemini API
▼

찾은 조항 내용을 바탕으로 출처를 명시하며 대화체로 답변 생성. 추가 정보를 제공하면 더 정확한 안내가 가능함을 언급.

출처 명시: 복리후생 > 휴가제도 제5조에 따르면...
친근한 대화체, 100~200자 내외
후속 질문 유도 멘트 포함
2-e 후속 질문 재탐색 판단
Gemini API
▼

사용자가 추가 정보를 제공했을 때, 기존 찾은 내용으로 답변 가능한지 판단. 필요하면 2-a부터 재탐색.

재탐색 불필요 → 기존 내용으로 더 구체적 답변
재탐색 필요 → 2-a로 돌아가 새 우선순위 계산
2-f Fallback
▼

모든 관련 카테고리를 탐색했음에도 답변을 찾지 못한 경우. 탐색한 항목을 투명하게 공개하고 담당 부서 문의를 안내.

"현재 등록된 규정에서 관련 내용을 찾을 수 없었습니다."
탐색한 카테고리 > 소카테고리 목록 나열
